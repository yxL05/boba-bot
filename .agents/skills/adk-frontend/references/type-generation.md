# Type Generation from ADK Bot

## Overview

When building frontend applications that interact with your ADK bot, you need access to strongly-typed definitions for your bot's actions, tables, and workflows. The ADK automatically generates TypeScript definition files during development and build, which your frontend can reference using TypeScript's triple-slash directives.

This approach ensures full type safety between your bot and frontend, catching errors at compile time rather than runtime.

---

## How ADK Generates Types

### Generation Process

When you run `adk dev` or `adk build`, the ADK analyzes your bot's source code and generates TypeScript definition files in the `.adk/` directory:

```bash
# Start development mode with hot reloading
adk dev

# Or build for production
adk build
```

### Generated Files

The `.adk/` directory contains multiple generated type definition files:

```
bot/
├── .adk/                          # Generated by adk dev/build (gitignored)
│   ├── action-types.d.ts          # Bot action definitions
│   ├── table-types.d.ts           # Table schema definitions
│   ├── workflow-types.d.ts        # Workflow input/output types
│   ├── conversation-types.d.ts    # Conversation handler types
│   ├── event-types.d.ts           # Event type definitions
│   ├── trigger-types.d.ts         # Trigger definitions
│   ├── state-types.d.ts           # State schema types
│   ├── integration-action-types.d.ts  # Integration action types
│   └── runtime.d.ts               # Runtime type definitions
├── src/
│   ├── actions/                   # Your action definitions
│   ├── tables/                    # Your table definitions
│   └── workflows/                 # Your workflow definitions
└── agent.config.ts
```

**IMPORTANT**: The `.adk/` directory is typically gitignored since these files are generated. Your frontend must reference them from the bot's `.adk/` directory using relative paths.

Defines types for all bot actions:

```typescript
declare module "@botpress/runtime/_types/actions" {

    (input: I): Promise<O>;

    sendMessage: {
      input: {
        agentId: string;
        ticketId: string;
        body: string;
        messageType: "comment" | "note";
        attachmentUrls?: string[];
        attachmentFiles?: Array<{
          content_type: string;
          data: string;

        }>;
      };
      output: {
        messageId: string;
        content: string;
        // ... other fields
      };
    };

    [K in keyof BotActionDefinitions]: CallableAction<
      BotActionDefinitions[K]["input"],

```

## TypeScript Triple-Slash References

### What Are Triple-Slash Directives?

Triple-slash directives are special TypeScript comments that tell the compiler to include type definitions from external files. They must appear at the very top of your TypeScript file.

```typescript
/// <reference path="../relative/path/to/types.d.ts" />
```

### Referencing Bot Types

Your frontend needs to reference the generated type files from your bot's `.adk/` directory:

```typescript
/// <reference path="../../../bot/.adk/table-types.d.ts" />
/// <reference path="../../../bot/.adk/workflow-types.d.ts" />
/// <reference path="../../../bot/.adk/action-types.d.ts" />
```

**Path Resolution**:
- Paths are relative to the current file
- Example: `packages/frontend/src/types/index.ts` → `packages/bot/.adk/`
- Use `../` to navigate up directories: `../../../bot/.adk/table-types.d.ts`

### Why Triple-Slash vs Import?

Triple-slash directives are used instead of imports because:

1. **Module Augmentation**: The generated files use `declare module` to augment existing modules
2. **No Runtime Code**: These are pure type definitions with no runtime impact
3. **Path Flexibility**: You can reference files outside your package's `node_modules`

---

## Importing Generated Types

After referencing the definition files, you can import the generated types:

```typescript
import type { BotActionDefinitions } from "@botpress/runtime/_types/actions";
import type { TableDefinitions } from "@botpress/runtime/_types/tables";
import type { WorkflowDefinitions } from "@botpress/runtime/_types/workflows";
```

These imports pull in the type definitions that were declared in the `.d.ts` files via triple-slash references.

## Creating Type Aliases

### Why Create Type Aliases?

Instead of writing verbose type expressions throughout your codebase:

```typescript
// Without aliases - verbose and error-prone
function sendMessage(
  input: BotActionDefinitions["sendMessage"]["input"]
): Promise<BotActionDefinitions["sendMessage"]["output"]> {
  // ...
}
```

You can create clean, reusable type aliases:

```typescript
// With aliases - clean and maintainable
export type SendMessageAction = BotActionDefinitions["sendMessage"];

function sendMessage(
  input: SendMessageAction["input"]
): Promise<SendMessageAction["output"]> {
  // ...
}
```

### Table Row Types

Extract specific table row types:

```typescript
// Table output types (includes id, createdAt, updatedAt)
export type TicketTableRow = TableDefinitions["TicketsTable"]["Output"];
export type MessageTableRow = TableDefinitions["MessagesTable"]["Output"];
export type UserTableRow = TableDefinitions["UsersTable"]["Output"];
export type AgentTableRow = TableDefinitions["AgentsTable"]["Output"];

// Table input types (for insertions/updates)
export type TicketInput = TableDefinitions["TicketsTable"]["Input"];
```

### Action Types

Extract action input and output types:

```typescript
export type SendMessageAction = BotActionDefinitions["sendMessage"];
export type AssignTicketAction = BotActionDefinitions["assignTicket"];
export type SnoozeTicketAction = BotActionDefinitions["snoozeTicket"];
export type OpenTicketAction = BotActionDefinitions["openTicket"];
export type CloseTicketAction = BotActionDefinitions["closeTicket"];

// Usage in service
async function sendMessage(input: SendMessageAction["input"]) {
  const result: SendMessageAction["output"] = await client.callAction({
    type: "sendMessage",
    input,
  });
  return result;
}
```

### Workflow Types

Extract workflow definitions:

```typescript
export type SuggestionWorkflow = WorkflowDefinitions["suggestKnowledge"];

// Usage
async function startSuggestion(input: SuggestionWorkflow["input"]) {
  const execution: SuggestionWorkflow["output"] = await client.createWorkflowExecution({
    workflow: "suggestKnowledge",
    input,
  });
  return execution;
}
```

### Advanced Type Manipulation

Sometimes you need to customize generated types:

```typescript
type InsightsField = WorkflowDefinitions["insights_extract_all"]["output"]["insights"][number];

// Override InsightsTableRow with properly typed insight field
export type InsightsTableRow = Omit<TableDefinitions["InsightsTable"]["Output"], "insight"> & {
  insight: InsightsField;
};
```

This pattern:
1. Extracts a specific field type from a workflow output
2. Uses `Omit` to remove the original field from the table type
3. Uses intersection (`&`) to add back the field with the correct type

## Complete Example: types/index.ts

Here's a complete type definition file showing all patterns in action:

```typescript
/// <reference path="../../../bot/.adk/table-types.d.ts" />
/// <reference path="../../../bot/.adk/workflow-types.d.ts" />
/// <reference path="../../../bot/.adk/action-types.d.ts" />
import type { Client } from "@botpress/client";
import type { BotActionDefinitions } from "@botpress/runtime/_types/actions";
import type { TableDefinitions } from "@botpress/runtime/_types/tables";
import type { WorkflowDefinitions } from "@botpress/runtime/_types/workflows";

// Table row types
export type TicketTableRow = TableDefinitions["TicketsTable"]["Output"];
export type MessageTableRow = TableDefinitions["MessagesTable"]["Output"];
export type UserTableRow = TableDefinitions["UsersTable"]["Output"];
export type AgentTableRow = TableDefinitions["AgentsTable"]["Output"];
export type WidgetTableRow = TableDefinitions["WidgetsTable"]["Output"];

// Advanced type manipulation - extract array element type from workflow
type InsightsField = WorkflowDefinitions["insights_extract_all"]["output"]["insights"][number];
// Override InsightsTableRow with properly typed insight field
export type InsightsTableRow = Omit<TableDefinitions["InsightsTable"]["Output"], "insight"> & {
  insight: InsightsField;
};

// Utility type to remove index signatures
type RemoveIndexSignature<T> = {
  [K in keyof T as string extends K ? never : number extends K ? never : K]: T[K];
};

// Base table row type (useful for generic table operations)
export type BaseTableRow = RemoveIndexSignature<Awaited<ReturnType<Client["findTableRows"]>>["rows"][number]>;

// Workflow types
export type SuggestionWorkflow = WorkflowDefinitions["suggestKnowledge"];

// Custom type for report file structure (not generated)
export type InsightsReport = {
  topics: Array<{
    topic: string;
    count: number;
    items: unknown[];
    coverage: "full_coverage" | "partial_coverage" | "no_coverage";
    evaluation: {
      completeness: number;
      relevance: number;
      length: number;
      usefulness: number;
      total: number;
    };
    answer: string;
    gaps?: string[];
    citations: Array<{
      idx: number;
      file: string;
      metadata: Record<string, unknown>;
      content: string;
    }>;
  }>;
};

// Action types
export type SendMessageAction = BotActionDefinitions["sendMessage"];
export type AssignTicketAction = BotActionDefinitions["assignTicket"];
export type SnoozeTicketAction = BotActionDefinitions["snoozeTicket"];
export type OpenTicketAction = BotActionDefinitions["openTicket"];
export type CloseTicketAction = BotActionDefinitions["closeTicket"];
export type UpdateTicketTagsAction = BotActionDefinitions["updateTicketTags"];
```

## Using Types in Services

### Bot Service Example

Here's how the types flow from bot → types → service:

**1. Bot Definition** (`packages/bot/src/actions/send-message.ts`):
```typescript
import { Action, z } from "@botpress/runtime";
import { MessageSchema } from "../schemas";

export const SendMessage = new Action({
  name: "sendMessage",
  description: "Sends a message to a user",
  input: z.object({
    agentId: z.string().describe("The ID of the agent sending the message"),
    ticketId: z
      .string()
      .describe("The ID of the ticket"),
    body: z.string().describe("The content of the message to be sent"),
    messageType: z
      .enum(["comment", "note"])
      .describe("The type of message to send")
      .default("comment"),
    attachmentUrls: z
      .array(z.string().url())
      .max(10)
      .optional(),
  }),
  output: MessageSchema,
  handler: async ({ agentId, body, ticketId, messageType }) => {
    // Implementation
  },
});
```

**2. Generated Type** (`.adk/action-types.d.ts` - auto-generated):
```typescript
declare module "@botpress/runtime/_types/actions" {
  export type BotActionDefinitions = {
    sendMessage: {
      input: {
        agentId: string;
        ticketId: string;
        body: string;
        messageType: "comment" | "note";
        attachmentUrls?: string[];
      };
      output: {
        messageId: string;
        content: string;
        // ... other message fields
      };
    };
  };
}
```

**3. Type Alias** (`packages/frontend/src/types/index.ts`):
```typescript
/// <reference path="../../../bot/.adk/action-types.d.ts" />
import type { BotActionDefinitions } from "@botpress/runtime/_types/actions";

export type SendMessageAction = BotActionDefinitions["sendMessage"];
```

**4. Service Implementation** (`packages/frontend/src/services/bot-service.ts`):
```typescript
import { Client } from "@botpress/client";
import type { SendMessageAction } from "../types";

class BotService {
  constructor(private client: Client) {}

  async sendMessage(
    input: SendMessageAction["input"]
  ): Promise<SendMessageAction["output"]> {
    const result = await this.client.callAction({
      type: "sendMessage",
      input,
    });
    return result.output;
  }
}
```

**5. Usage in Component** (with full type safety):
```typescript
import { botService } from "../services/bot-service";

async function handleSendMessage() {
  // TypeScript knows exactly what fields are required
  const message = await botService.sendMessage({
    agentId: "agent_123",
    ticketId: "ticket_456",
    body: "Hello, how can I help you?",
    messageType: "comment", // TypeScript enforces "comment" | "note"
    // TypeScript will error if we forget required fields
  });

  // TypeScript knows the shape of the returned message
  console.log(message.messageId);
  console.log(message.content);
}
```

## Keeping Types in Sync

### Automatic Updates with `adk dev`

The `adk dev` command watches your bot's source files and regenerates types automatically:

```bash
cd packages/bot
adk dev
```

When you modify an action, table, or workflow:

1. Save the file
2. ADK detects the change
3. Types are regenerated in `.adk/`
4. Your frontend's TypeScript server picks up the changes
5. Your IDE shows updated types immediately

### Handling Breaking Changes

When you modify bot definitions, your frontend may show type errors:

```typescript
// You changed the action to require a new field
const message = await botService.sendMessage({
  agentId: "agent_123",
  ticketId: "ticket_456",
  body: "Hello",
  messageType: "comment",
  // ERROR: Property 'priority' is missing
});
```

**This is good!** TypeScript is telling you exactly what needs to be updated to match the new bot contract.

### Build-Time Type Checking

Add a build step that checks your bot types are up-to-date:

```json
{
  "scripts": {
    "build": "pnpm run build:bot && pnpm run build:frontend",
    "build:bot": "cd packages/bot && adk build",
    "build:frontend": "cd packages/frontend && tsc --noEmit && vite build"
  }
}
```

This ensures:
1. Bot types are generated before frontend build
2. TypeScript checks all types match
3. Build fails if frontend uses outdated types

## Troubleshooting

### Cannot Find Module '@botpress/runtime/_types/actions'

**Problem**: TypeScript can't find the generated type modules.

**Solutions**:

1. **Verify triple-slash references** at the top of your file:
   ```typescript
   /// <reference path="../../../bot/.adk/action-types.d.ts" />
   ```

2. **Check the relative path** is correct:
   ```bash
   # From packages/frontend/src/types/index.ts
   # To packages/bot/.adk/action-types.d.ts
   # Path should be: ../../../bot/.adk/action-types.d.ts
   ```

3. **Ensure `.adk/` directory exists**:
   ```bash
   cd packages/bot
   adk dev  # or adk build
   ls .adk/  # Should list generated files
   ```

4. **Restart TypeScript server** in your IDE:
   - VS Code: `Cmd+Shift+P` → "TypeScript: Restart TS Server"

### Path Issues in Monorepo

If you have a complex monorepo structure:

```
my-project/
├── apps/
│   └── frontend/
│       └── src/
│           └── types/
│               └── index.ts
└── packages/
    └── bot/
        └── .adk/
```

Your path from `apps/frontend/src/types/index.ts` to `packages/bot/.adk/` would be:

```typescript
/// <reference path="../../../../../packages/bot/.adk/action-types.d.ts" />
```

**Tip**: Use your IDE's file explorer to verify the path by counting directory levels.

### Types Not Updating

If types seem stale after changing your bot:

1. **Restart `adk dev`**:
   ```bash
   cd packages/bot
   # Stop adk dev (Ctrl+C)
   adk dev
   ```

2. **Force rebuild**:
   ```bash
   cd packages/bot
   rm -rf .adk/
   adk build
   ```

3. **Restart TypeScript server** in your IDE

4. **Check for TypeScript errors** in the bot itself - ADK won't generate types if there are compilation errors

### Missing Generated Files

If `.adk/` directory is empty or missing files:

1. **Check bot builds successfully**:
   ```bash
   cd packages/bot
   adk build
   ```

2. **Verify file structure**:
   ```bash
   ls -la src/actions/
   ls -la src/tables/
   ls -la src/workflows/
   ```

3. **Check for syntax errors** in your action/table/workflow definitions

4. **Ensure proper exports**:
   ```typescript
   // Good - properly exported
   export const MyAction = new Action({ ... });

   // Bad - not exported
   const MyAction = new Action({ ... });
   ```

### Git Merge Conflicts

Since `.adk/` is gitignored, you won't have merge conflicts in generated files. However:

1. After pulling changes, **regenerate types**:
   ```bash
   cd packages/bot
   adk build
   ```

2. **Fix any type errors** in your frontend that result from bot changes

## Best Practices

### 1. Centralize Type Definitions

Create a single `types/index.ts` file that exports all bot-related types:

```typescript
/// <reference path="../../../bot/.adk/table-types.d.ts" />
/// <reference path="../../../bot/.adk/workflow-types.d.ts" />
/// <reference path="../../../bot/.adk/action-types.d.ts" />

import type { BotActionDefinitions } from "@botpress/runtime/_types/actions";
import type { TableDefinitions } from "@botpress/runtime/_types/tables";
import type { WorkflowDefinitions } from "@botpress/runtime/_types/workflows";

// Export all action types
export type SendMessageAction = BotActionDefinitions["sendMessage"];
// ... etc

// Export all table types
export type TicketTableRow = TableDefinitions["TicketsTable"]["Output"];
// ... etc

// Export all workflow types
export type SuggestionWorkflow = WorkflowDefinitions["suggestKnowledge"];
// ... etc
```

Then import from this single location throughout your codebase:

```typescript
import type { SendMessageAction, ConversationTableRow } from "../types";
```

### 2. Use Descriptive Type Names

```typescript
// Good - clearly indicates purpose
export type SendMessageAction = BotActionDefinitions["sendMessage"];
export type MessageTableRow = TableDefinitions["MessagesTable"]["Output"];

// Bad - unclear abbreviations
export type SMA = BotActionDefinitions["sendMessage"];
export type MTR = TableDefinitions["MessagesTable"]["Output"];
```

### 3. Document Custom Type Manipulations

When creating complex types, add comments explaining the logic:

```typescript
// Extract the array element type from the workflow output
// This ensures the table's insight field matches the workflow's output exactly
type InsightsField = WorkflowDefinitions["insights_extract_all"]["output"]["insights"][number];

// Override the table's insight field type to match the workflow
export type InsightsTableRow = Omit<TableDefinitions["InsightsTable"]["Output"], "insight"> & {
  insight: InsightsField;
};
```

### 4. Keep Frontend and Bot in Same Workspace

In a monorepo, keep your bot and frontend packages together:

```
my-project/
├── packages/
│   ├── bot/          # ADK bot
│   └── frontend/     # Frontend app
└── pnpm-workspace.yaml
```

This ensures:
- Shorter relative paths for triple-slash references
- Both can be built in a single command
- Changes are tested together

### 5. Add Type Checking to CI/CD

```yaml
# .github/workflows/ci.yml
- name: Build bot
  run: cd packages/bot && adk build

- name: Type check frontend
  run: cd packages/frontend && tsc --noEmit

- name: Build frontend
  run: cd packages/frontend && pnpm build
```

## Summary

**Key Takeaways**:

1. **ADK auto-generates types** in `.adk/` during `adk dev` or `adk build`
2. **Use triple-slash references** to include generated types in your frontend
3. **Create type aliases** for cleaner, more maintainable code
4. **Types update automatically** when using `adk dev` watch mode
5. **Path resolution** is critical - verify relative paths carefully
6. **Centralize type definitions** in a single file for consistency
7. **TypeScript catches breaking changes** between bot and frontend at compile time

By following these patterns, you'll maintain complete type safety between your ADK bot and frontend application, catching integration issues during development rather than at runtime.
